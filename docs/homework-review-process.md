# 作业批改记录规范

## 固定要求

以后每次批改作业，都必须在本地仓库里记录一份批改记录。

批改记录统一放在 `reviews/` 目录下，文件名格式：

- `YYYY-MM-DD-<topic>-homework-review.md`

例如：

- `2026-06-24-explain-analyze-homework-review.md`

## 每次批改必须记录的内容

每份批改记录至少包含：

- 批改对象：对应的练习文件或笔记文件。
- 验证环境：PostgreSQL / Greenplum 版本、数据库名、schema、数据量。
- 验证方式：是否实际运行 SQL，是否只读验证，是否创建临时 schema。
- 正确答案或参考结果。
- 学员答案中正确的地方。
- 需要修正的地方。
- 学员提出的问题和回答。
- 当前掌握程度评价。
- 下一步练习建议。

## 数据库验证原则

如果需要在本地数据库运行验证 SQL：

- 优先使用临时 schema，避免污染已有作业表。
- 不直接删除或覆盖 `public` schema 里的作业数据，除非用户明确要求。
- 验证完成后清理临时 schema。
- 如果只需要复核执行计划，优先使用只读 `EXPLAIN` 或 `EXPLAIN ANALYZE`。

## Git 处理原则

批改记录可以单独提交。

如果学员的作业文件还有未提交改动：

- 不把作业文件混进批改记录提交。
- 只提交 `reviews/` 里的批改记录和必要的 README 入口。
- 在最终回复里说明还有哪些作业文件未提交。

## 批改口径

批改时重点看：

- 是否能把执行计划节点归属到正确表或步骤。
- 是否区分估算值和实际值。
- 是否正确理解过滤条件、JOIN、聚合、排序和 Buffers。
- 是否能说明索引为什么被使用或为什么没有被使用。
- 是否能形成自己的结论，而不是只抄执行计划。
